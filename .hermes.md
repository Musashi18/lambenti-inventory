# Lambenti Inventory — Hermes Multi-LLM Operating Mode

This project runs under the Lambenti profile's two-tier orchestration default.

## Controller / worker split

- Hermes/GPT-5.5 is the interactive controller and senior reviewer.
- Local Qwen2.5-Coder-7B-Instruct in LM Studio is the default `delegate_task` leaf worker for low-risk, bounded work.
- Hermes owns routing, tool execution, validation, escalation, and final synthesis.
- Delegate outputs are advisory until Hermes verifies them from files, commands, tests, browser output, or database state.

## Default routing

Use local Qwen delegates for:

- read-only repo audits and file summaries;
- small code-draft proposals before Hermes edits;
- targeted test-failure triage;
- documentation/checklist drafts;
- independent low-risk review of narrow diffs;
- private/local context summarization that should not be sent remotely.

Keep Hermes/GPT-5.5 in direct control for:

- Prisma schema migrations, database mutations, stock ledger behavior, purchase receiving, invoices, journals, payments, landed cost, and accounting correctness;
- security, auth, credentials, payments, deployment, destructive commands, or production-critical work;
- ambiguous root-cause debugging after a local attempt fails;
- final acceptance/rejection of code, architecture, and high-risk product decisions.

## Delegation discipline

- Pass bounded context; do not dump the whole repo into a delegate.
- Pass narrow toolsets where possible (`file`, `terminal`, `search`, or `web` only when needed).
- Prefer two or fewer concurrent local Qwen children because LM Studio is tuned here for 32,768 context and `parallel=2`.
- Do not let worker models directly authorize side effects. They propose; Hermes classifies risk, executes, and verifies.
- Never send secrets, private customer data, API keys, browser cookies, SSH keys, database URLs, or raw proprietary dumps to remote models without explicit user authorization.

## Lambenti app invariants

- The database is the source of truth.
- Email/order imports may create ordered PO lines but must not receive physical stock.
- Accounting/invoices/journals must not mutate physical inventory.
- Physical stock changes require explicit receiving or stock-movement actions.
- Cost/accounting changes must preserve provenance and review/apply boundaries.

## Required startup context

Before planning substantial work, inspect:

- `AGENTS.md`
- `HERMES_STATE.md`
- `TASK_QUEUE.md`
- `DECISIONS.md`
- `ISSUES.md`
- `git status --short`

For app verification, use the current repo gates:

```bash
npm run lint
npm run typecheck
npm run test:serial
npm run build
```

For UI/server-action changes, also run a production smoke at `http://127.0.0.1:5173` with `npm run start:local` and inspect browser console errors.
