# DECISIONS

- Tracking-only Alibaba portal message/tracking evidence belongs in shipment-tracking workflows, not the active Order Email Agent queue. `/integrations/email-import` and reassessment should exclude portal snapshots with message-thread/tracking-only subjects or Message Center URLs so actionable supplier emails keep their item/supplier matches visible; tracking metadata remains available through `/tracking`.
- Ship24 is the default Lambenti tracking-status provider. Use `LAMBENTI_TRACKING_STATUS_PROVIDER=SHIP24`, a Ship24 dashboard API key in `LAMBENTI_TRACKING_STATUS_AUTH_TOKEN`, `LAMBENTI_TRACKING_SHIP24_BASE_URL=https://api.ship24.com`, destination `CA`, and the existing scheduler hook. Keep `CUSTOM_HTTP`/`LAMBENTI_TRACKING_STATUS_URL_TEMPLATE` as an escape hatch. The selection is based on current official evidence for global parcel API/webhook tracking, China Post/UPS coverage, courier auto-detection, and low-volume pricing fit.
- The `/tracking` Capture Alibaba action must run the Alibaba portal agent in tracking-only mode before saved-evidence backfill. Tracking-only mode uses the dedicated non-default Alibaba automation Chrome profile by default because current Chrome blocks controlled DevTools scraping of the normal/default Work profile; only opt back into Work-profile control with explicit override envs after validating Chrome allows it. Normal Capture should allow safe saved-login assist (`Continue as Musashi` / trusted Alibaba account confirmation), use the Lambenti email as trusted-account confirmation fallback when the dedicated profile lacks Chrome account metadata, and, if login/security is still required, reopen a manual Chrome handoff instead of silently closing. Portal imports must force `autoApply:false` and `autoCreateInvoices:false`; Capture may save/link logistics evidence only. The required live-check sequence is Alibaba orders Delivering tab, Alibaba orders Completed & In Review tab, then individual Alibaba Message Center supplier conversations. Tracking-only scans should be stale-aware: skip order items marked `Waiting for supplier to ship`, skip order candidates once tracking has already been captured for that item or already exists in persisted `TrackingNumber` rows, hydrate local memory from saved tracking rows before clicking Alibaba order candidates, remember unchanged no-tracking/null order attempts by stable candidate key/fingerprint so they do not get reopened forever, keep `order:<id>` memory aligned to the candidate order id even when the opened/list page contains other order numbers, remember each read chat by stable key/list fingerprint, skip unchanged chats on later runs, and keep reading other new chats after one chat yields tracking. Completed & In Review candidate selection should prioritize/dedupe read-only `Track Package` / `Track Shipment(s)` / logistics controls, ignore generic Alibaba logistics-services marketplace links, and use enough first-pass budget to cover previous orders without sacrificing future stale skips. The memory is local runtime state under `var/alibaba-portal-agent-state.json` and must be updated each run. Tracking-only scans should remain bounded for operator latency unless explicitly overridden (`maxCandidates=4`, `maxMessageThreads=12`, `messageListScrolls=4`), but keep the server action timeout long enough for the current full sweep plus import upload.
- Alibaba Message Center capture should open `https://message.alibaba.com/message/messenger.htm`, read each visible supplier conversation/thread section individually, save only shipment/tracking-relevant `conversationContext`, and avoid composer/send/reply/inquiry controls. Full controlled scraping can require the Work Chrome profile to be closed or manual CAPTCHA/2FA/security handoff to be completed first.
- Delivered tracking is retained as shipment history, not active work: delivered rows are removed from the `/tracking` active refresh list/table, shown in delivered history with linked evidence and total ship time, and should have `nextRefreshAt` cleared. Ship time is measured from earliest carrier event to delivered timestamp when available; do not invent a duration when carrier event range is missing.
- Tracking numbers are evidence/logistics metadata, not receiving proof: tracking capture/refresh may link to purchase orders/imports and store shipment events, but it must not create stock lots, stock movements, received quantities, invoice approval, payment, or delivery confirmation.
- Tracking-service refresh defaults to Ship24 via `LAMBENTI_TRACKING_STATUS_PROVIDER=SHIP24` and `LAMBENTI_TRACKING_STATUS_AUTH_TOKEN`; the older `LAMBENTI_TRACKING_STATUS_URL_TEMPLATE` custom-HTTP path remains available for fallback providers. Without a configured provider token/URL the app marks tracking rows `CONFIG_REQUIRED` and keeps the UI useful for evidence/linking rather than fabricating carrier status. Scheduled/background refresh should use the same `/api/tracking/refresh` path through `npm run agent:tracking-refresh` or an equivalent scheduler hook, and must stay shipment-metadata-only.
- Alibaba tracking capture is scoped to recent evidence: portal/saved-evidence tracking extraction should skip and prune Alibaba portal tracking rows whose linked order/evidence date is older than 3 months. Recent completed-order tracking may be linked to existing PO/import records; old tracking evidence should not remain active on `/tracking`.
- Tracking extraction should be context-aware: accept strong carrier formats and numeric IDs near tracking/logistics/waybill context, exclude the source Alibaba order ID, and avoid phone/product/order false positives.
- Email/import tracking backfill may scan archived evidence because archived order-agent imports can still contain real logistics numbers; it must preserve source provenance such as `SYNCED_EMAIL` instead of forcing all tracking rows to Alibaba portal source.
- Gitignored runtime/browser-profile artifacts under `var/**` are not source/test files; ESLint and Vitest should ignore them so Alibaba profile captures do not break quality gates.
- Email order imports and reassessment may update item cost/provenance from supplier evidence, but they must not overwrite an item `preferredSupplierId` that was already manually selected; the order supplier is only a fallback for items with no preferred supplier.
- Integration tests that exercise real seed/catalog SKUs must snapshot and restore protected item master-data fields or use test-only items; cleanup must not delete temporary suppliers that are still referenced by real item rows.
- Prefer a Receiving workbench over a new schema-heavy module for this pass because it closes the ordered-PO → human receipt → stock ledger loop with low risk.
- Reuse `receivePurchaseOrderLine()` rather than adding separate receiving logic; inventory changes must remain ledger-based and human-confirmed.
- Do not auto-receive stock from order emails or invoices; the `/incoming` UI copy explicitly reinforces this boundary.
- Approved purchase requests convert to purchase orders through an explicit human purchasing action that creates `DRAFT` POs only; conversion must not mark ordered, receive stock, create stock lots/movements, or bypass `/incoming` receiving.
- Purchase-request conversion requires supplier and unit-price evidence (`targetUnitPrice` or item `estimatedUnitCost`) so draft POs are reviewable purchasing records rather than hidden zero-price placeholders.
- Do not add a Prisma migration for this pass; the existing PO line, stock lot, stock movement, and audit models support the workflow.
- Use inline action state for receiving failures so operator mistakes (over-receipt, duplicate lot, permission failure) do not crash production pages.
- Supplier dropdown options should be sourced from the same active, displayable, deduped supplier profiles shown on `/suppliers`, not from `confirmedByHuman` alone.
- `confirmedByHuman` now means a human-confirmed supplier record, not dropdown eligibility; active displayable suppliers are dropdown-eligible unless archived or filtered as junk/import noise.
- A custom supplier typed by an operator is allowed because it is human-provided; create it with `companyName = name`, `confirmedByHuman = true`, conservative purchasing defaults, and an audit log. This must not mutate stock quantities.
- Dashboard assembled-package total is specifically the ledger-derived on-hand quantity for the Lambenti assembled package finished good (`LAMBENTI_PACKAGE`), not a sum of every active finished good or subassembly.
- Dashboard build capacity is scoped to active BOMs for `LAMBENTI_PACKAGE`; competing finished-good/subassembly BOMs must not determine the package build card.
- Dashboard build-capability logic uses summed `BOMLine.quantity` values and keeps valid zero-capacity package BOMs visible internally, but the dashboard card should hide the summed BOM quantity and show only the package SKU plus bottleneck.
- Accounting source documents are their own auditable record type (`AccountingDocument`) because serious accounting needs immutable original file/email evidence, SHA-256 idempotency, extracted text, structured analysis, status, and links separate from invoices/orders.
- Upload/analyze of accounting documents is evidence-only: do not create suppliers, purchase orders, invoices, payments, stock lots, or stock movements until an explicit human apply action.
- Applying a reviewed supplier invoice document may create/update `SupplierInvoice` provenance and source-derived invoice totals/dates, but must not receive stock; receiving remains `/incoming` only.
- Keep source files private under `var/accounting-documents` and expose retrieval through an authenticated accounting download route, not public static paths.
- Prefer root `/accounting` as the operator accounting workbench, while `/accounting/invoices` remains the narrower invoice ledger/action page.
- For Lambenti app verification, prefer lint, typecheck, serial Vitest (`npm run test -- --run --no-file-parallelism` or `npm run test:serial`), and build before final readiness claims; use browser/runtime smoke for UI/server-action changes.
- Alibaba graphical product-card parser output represents explicit supplier line boundaries. Within that extraction batch, keep distinct card segments as separate editable lines even if fuzzy duplicate detection would otherwise merge matching quantity/price and overlapping product wording.
- For broad or continuation Lambenti work, future agents should load `lambenti-parallel-engineering-operator`, inspect checkpoint files, use session_search for continuity, and load only task-relevant references.
- For the Windows Lambenti CLI profile, keep core engineering tools enabled but disable clearly unusable profile bloat such as macOS-only `computer_use`.
- Hermes/Lambenti two-tier orchestration is the default for this repo: Hermes/GPT-5.5 stays controller/senior reviewer; low-risk bounded delegates use local LM Studio Qwen; delegates are leaf/advisory and all side effects must still be executed/verified by Hermes.
- With Qwen2.5-Coder-7B at 32,768 context, keep LM Studio `parallel=2` and Hermes delegation `max_concurrent_children=2` for stable local throughput instead of trying to run Qwen as the primary Hermes tool-heavy model.
- A running Hermes GUI session may cache delegation config from startup; after changing `delegation.*` defaults, restart/open a new session before assuming `delegate_task` uses the new local worker.
- Uploaded accounting documents that produce no extractable text must stay as reviewable source evidence with retry/manual re-analysis paths; do not leave saved source files as dead-end `FAILED` rows when an operator can retry OCR or paste text.
- Accounting OCR is best-effort evidence extraction. OCR warnings and pro forma documents remain review-gated; payment receipts may be analyzed and attached as evidence, but they must not post AP, mark invoices paid, or receive stock without explicit human apply/payment/receiving actions.
- Multi-document accounting bundles use `AccountingDocument` links/status (`ATTACHED`) rather than overwriting `SupplierInvoice.sourceDocumentPath/sourceDocumentHash`; receipts, packing slips, customs documents, quotes, and order notices attach as evidence only and must not create AP, mark paid, or receive stock.
- Supplier invoice numbers are unique per supplier via normalized `invoiceNumberKey`, not globally; the same supplier invoice number across different suppliers is allowed, while same-supplier case/spacing variants dedupe to one payable.
- Purchase orders may have multiple supplier invoices for staged supplier payments (deposit/partial/final/credit-note style records); accounting automation may create distinct invoices only when explicit invoice/source identity exists, and still must not receive stock.
- Payment reconciliation is an explicit human accounting step using imported bank transactions/payment allocations; OCR payment receipts remain evidence-only and do not mark invoices paid.
- Posted accounting journals are now implemented for AP invoice approval and AP payment reconciliation. AP invoice/payment posting must be source-linked, idempotent, balanced, and transactional; missing GL mappings block the human approval/reconciliation action instead of creating bad entries.
- GL account snapshots on journal lines preserve historical account code/name/type even if the chart of accounts changes later.
- Bank imports, OCR/payment receipts, source-document uploads, and invoice creation remain non-posting until a human approval/reconciliation action occurs; journal posting must not receive stock or modify PO received quantities.
- Customer/AR journal posting is still future scope; current AR records track customer invoices/status/payment references without posting revenue, receivable, tax payable, or cash journals.
- Landed-cost reports are read-only allocations from invoice fields and invoice lines; recoverable GST/HST is excluded from inventory landed cost, and reports must not mutate stock lots, stock movements, item costs, or PO received quantities.
- Customer invoices / AR are customer-facing accounting records only. Creating/sending/marking AR invoices paid must not consume stock, reserve inventory, ship products, or bypass fulfillment/receiving workflows; paid AR requires an explicit payment reference.
- Dashboard email import catalog-matching notices are intentionally removed from the dashboard human-approval queue; unresolved email/import line matching belongs in Order Agent/accounting/invoice workflows, not the high-level operations dashboard.
- Finished goods and internal subassemblies (`FINISHED_GOOD`) are assembled internally and must not be suggested for purchase requests, purchase recommendations, or automation reorder findings; component shortages remain purchasable.
- Alibaba portal automation must use the signed-in Work/Lambenti Google Chrome profile when configured, and must not bypass slide CAPTCHA/security/2FA checks. The safe flow is headed Chrome human handoff, then session reuse for order/message/tracking extraction.
- Alibaba portal autonomous navigation is limited to read-only evidence surfaces: order details, message threads, invoices/receipts, payment details, shipment/tracking/logistics/waybill pages, and Trade Assurance detail links. The agent must exclude side-effect controls such as pay, place order, confirm receipt, release payment, send message, delete, refund/dispute, or logout.
- Source-document deletion from `/accounting` may remove the saved source file and `AccountingDocument` analysis row only through a deliberate accounting action; it must not mutate stock, PO receiving, paid status, or journal entries.
- `/accounting` should prioritize operator attention before feature discovery: show next-action/exception queues, document triage counts, and readiness badges ahead of raw source-document tables. This is UI-only guidance; it must not relax human gates for AP creation, invoice approval, payment reconciliation, journal posting, or physical receiving.
